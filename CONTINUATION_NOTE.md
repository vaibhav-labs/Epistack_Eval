# Continuation note: from a proven detector to the Epistack Eval Standard

*This note is for the funded-work conversation FLF raises with finalists. It states what is already built and de-risked, what I would build next, roughly what each stage costs in time and compute and the one research obstacle that gates the most valuable version.*

## What is already built and de-risked

I have a working detector that runs in one command, sends real reasoners through a base under a knowledge mask and reports whether a reasoner leaning on the base double-counts correlated evidence, misses the crux, or leans on withdrawn evidence. It has caught a real double-counting failure on a real, externally-built 26-claim base that had already been de-correlated by hand and it has done so reproducibly across fourteen runs. It also closes the loop in minimal form through Part D, which collapses the correlated cluster into one node and, on the stronger model, confirmed in all three runs that the repaired base lands at the warranted weight, while the check reproducibly flagged that a too-weak collapse overshot and that the same faithful collapse still undershoots on the weaker model. The base is an input, so the same instrument already runs unchanged on any base with no code editing. The risk that this is a mockup, or that the result is baked in, is therefore retired, because the reasoning comes from a real model and any reviewer can rerun it.

## Stage one, the Standard, which is where the compounding value is

The first thing I would build is the Epistack Eval Standard, which is a single linkage-integrity score that any base can be measured against, so that bases built by wholly different people and methods on the same question become comparable on whether their rigor survives a reasoner. This follows directly from the base being an input, because the instrument already runs across many bases without modification, so the work is not a new mechanism but a stable, aggregated score plus the harness to run it at breadth. The immediate deliverable is a scorecard that takes a set of bases on one question and ranks them by how well the reasoner they produce holds warranted calibration. I estimate this at roughly four to six weeks of build, with compute dominated by the number of bases times the number of probes times the runs needed for stability, which I would expect to sit in the low thousands of model calls per question rather than anything exotic.

## Stage two, the adversarial version, which is the sharpest score

The sharpest version of the Standard is adversarial, because it does not measure calibration once but measures how well a reasoner leaning on a base holds its calibration under sustained, targeted cross-examination and then ranks bases by which one produces a reasoner that decays under pressure. A base whose linkages are sound should hold and a base that only looked sound should come apart when pressed and this turns resolving a contested question into something measurable across competing structures. This is more open-ended, so I would scope it as a research build of roughly eight to twelve weeks, with compute rising with the depth of cross-examination per base.

## The obstacle that gates ranking, which I state plainly

There is a real problem between a detector and a ranking and I do not want to hide it. To rank bases against each other, the measurement has to be steadier than the differences between the bases and right now it is not, because the same base reads from 0.09 to 0.14 across runs. That deviation is fine when I report a single range, which is what I do now and it is not fine for ranking, because the gap between two bases can be smaller than the wobble inside one base. So the core research task of the comparative version is driving the measurement noise below the between-base signal and more runs narrow it slowly while a less coarse metric that aggregates over many probe points, rather than differencing two coarse outputs, could narrow it faster. This obstacle also gates the repair arm, because confirming that a repair actually dropped the double-counting needs a measurement stable enough to trust, so solving it unlocks both arms at once and I would treat it as the first research milestone rather than an afterthought.

## Stage three, repair at scale, which Part D already proves in miniature

The third arm is repair rather than only detection and ranking and Part D is its proof of concept, already run on the real base. The instrument already locates the correlation structure a reasoner is blind to, the repair reshapes the base by collapsing the correlated cluster into one node and the re-measurement confirmed on the stronger model that the repaired base lands at the warranted weight while catching that a too-weak collapse overshot and that the same fix undershoots on the weaker model, which is the model-dependence a scaled repair would have to handle. Scaling this means automating the hand-off, so that the cluster the detector identifies is piped straight into the repair with no manual step and then running the reshape-and-reconfirm loop across many clusters and bases. I would scope the automated hand-off at roughly three to four weeks once the measurement is stable, because the two halves already exist and the remaining work is connecting them and confirming the fix at breadth.

## What I would want from a continuation

I would want the compute to run the Standard across a real set of competing bases on at least one question, the time to solve the measurement-stability milestone first because it gates everything downstream and an ongoing line to FLF so that the score is shaped by the people who would use it to compare submissions and audit bases. What is built is the detector, the minimal repair loop and the evidence that the failure they catch is real and the continuation is the path from that proof point to a score that competing work can be measured against.
